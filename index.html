Why the simulation is built this way
Forces diversity within the cohort

A single "average user" simulation smooths over exactly the edge cases that matter most for accessibility testing, like a specific medical condition interacting with a specific UI element. Generating N distinct personas with varied secondary traits pushes the model to surface those.

One walkthrough call per persona

Each persona gets a dedicated model call for their step-by-step walkthrough. This keeps every walkthrough grounded in that persona's specific traits and produces trait-linked friction points ("small tap targets plus arthritis") instead of generic UX platitudes.

Synthesis separates signal from noise

A final pass splits common issues from outlier issues and ranks the design changes that would most reduce friction. That is usually what you actually want out of a simulation like this: where to prioritize fixes.

Built-in epistemic caveat

The caveat banner and the synthesis prompt both keep the output honest about being a hypothesis generator, not a replacement for real usability testing with the actual population. That matters more here than in most simulations, since real people's health and demographic traits are being modeled.

Suggested usage pattern

Run the simulation multiple times with different cohort sizes or slightly perturbed audience parameters (a narrower age band, an added condition) and compare the results. That is often more useful than a single large run, since it shows which findings are stable and which are parameter-sensitive.
